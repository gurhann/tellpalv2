# Write Flyway Migration

## Overview

Write forward-only Flyway migrations that fit a Spring Boot Modulith backend. Keep SQL valid for PostgreSQL 15, model tables in snake_case, and make schema decisions explicit so the user gets both runnable SQL and a short rationale.

## Workflow

1. Inspect the target module, existing migrations, and nearby aggregates before writing SQL.
2. Create a new file named `V{number}__description.sql`. Never rename or edit an applied migration unless the user explicitly asks for a repair strategy.
3. Model tables with explicit primary keys, foreign keys, `NOT NULL`, and check constraints where the domain has hard validation rules.
4. Add indexes for join keys, lookup fields, uniqueness, and columns likely to appear in `WHERE`, `ORDER BY`, or foreign-key joins.
5. Explain the schema decisions after the SQL, including constraints, indexes, and any tradeoffs.

## Authoring Rules

- Use PostgreSQL syntax only.
- Use lowercase snake_case for tables, columns, indexes, and constraints.
- Prefer `bigint generated by default as identity` for numeric primary keys unless the surrounding module already uses UUIDs.
- Declare `timestamptz` for audit timestamps when time zone correctness matters.
- Add `created_at` and `updated_at` only when the module already uses audit fields or the new table clearly needs them.
- Make nullable columns intentional. Default to `NOT NULL` for required domain data.
- Add foreign keys with explicit `references` clauses and choose `on delete` behavior deliberately.
- Use `check` constraints for enums, ranges, non-negative counts, and similar domain rules when the rule belongs in the database.
- Keep migrations idempotency-safe for Flyway by avoiding conditional DDL that can hide drift unless the user explicitly requests it.

## Output Format

Return all of the following:

1. The SQL migration file content.
2. A short explanation of the schema decisions.
3. A concise list of constraints and indexes that were added or changed.

## Safety Checks

- Ensure every new table has a primary key.
- Ensure new foreign-key columns have supporting indexes unless the table is tiny and clearly not queried that way.
- Check names against the bounded context and module terminology; do not leak cross-module concepts into the wrong schema.
- Prefer additive changes. For destructive changes, call out data migration and rollback risk explicitly.
- When altering existing tables, preserve backward compatibility for running application code unless the user asked for a breaking migration.
